[ISSUE]

**Title:** Unable to Register Route with Names Containing Both Dots and Colons

**Description:**
After merging branch '0.18', attempting to register a route with a name that includes both dots (`.`) and colons (`:`) results in a `ValueError`. The recent changes were intended to allow route names to be a sequence of Python identifiers separated by dots or colons, but this combination is still causing issues.

**Example Code:**
```python
from aiohttp.web import UrlDispatcher, PlainRoute

def handler(request):
    return 'Hello'

router = UrlDispatcher()

# Attempting to register a route with both dots and colons in the name
route = PlainRoute('GET', handler, 'test.test:test', '/handler/to/path')
router.register_route(route)
```

**Expected Behavior:**
Registering a route with a name like `'test.test:test'` should succeed without errors, as the name follows the updated rules allowing multiple identifiers separated by dots or colons.

**Actual Behavior:**
A `ValueError` is raised with the message:
```
ValueError: Incorrect route name value, Route name should be a sequence of python identifiers separated by dot or column
```
This prevents the registration of route names that include both dots and colons, contrary to the intended flexibility introduced in the recent commit.

[ISSUE]
**Title:** Registering Regex Patterns in Routes Causes AttributeError

**Description:**
When attempting to register a route using a regular expression, the application raises an `AttributeError` because it incorrectly assumes that the path is a string and tries to call the `startswith` method on it.

**Example Code:**
```python
import re
from aiohttp.web import UrlDispatcher, Request, Response

def make_handler():
    async def handler(request):
        return Response(text="Success")
    return handler

router = UrlDispatcher()
handler = make_handler()

# Attempting to add a route with a regex pattern
router.add_route('GET', re.compile('^/test/(.*)$'), handler)

# Creating a mock request
request = Request(app=None, message=None, payload=None, transport=None, reader=None, writer=None)
request.method = 'GET'
request.path = '/test/foo/bar'

# Resolving the request
info = await router.resolve(request)
```

**Expected Behavior:**
The route should be successfully added using the regex pattern, and resolving a request matching the pattern should return the appropriate handler without errors.

**Actual Behavior:**
An `AttributeError` is raised:
```
AttributeError: 're.Pattern' object has no attribute 'startswith'
```
This error occurs because the `add_route` method attempts to call `startswith` on a regex pattern object, which does not have this method.

[ISSUE]

**Title:** Incorrect `SERVER_PORT` in WSGI Environ Causes Test Failures

**Description:**
The `SERVER_PORT` variable in the WSGI environ is not correctly set to the TCP/IP port number on which the request is received from the client. Instead, it incorrectly uses the port specified in the `HOST` header, leading to inconsistencies.

For instance, when the `HOST` header includes a different port than the one the server is listening on, `SERVER_PORT` should reflect the actual listening port. However, the current implementation assigns `SERVER_PORT` based on the `HOST` header, causing discrepancies.

**Example Buggy Code:**
```python
def test_server_port():
    transport.get_extra_info.side_effect = [('1.2.3.4', 9999), ('3.4.5.6', 82)]
    headers.update({'HOST': 'example.com:9999'})
    environ = create_wsgi_environ()
    print(environ['SERVER_PORT'])  # Expected: '82'
```

**Expected Behavior:**
`SERVER_PORT` should be set to the actual TCP/IP port number on which the request is received (`82` in this case), regardless of the port specified in the `HOST` header.

**Actual Behavior:**
`SERVER_PORT` is incorrectly set to `'9999'`, the port from the `HOST` header, instead of the actual listening port `'82'`. This causes tests expecting the correct port to fail.

[/ISSUE]
